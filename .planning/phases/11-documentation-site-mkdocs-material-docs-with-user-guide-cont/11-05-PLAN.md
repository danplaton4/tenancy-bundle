**`docs/architecture/event-lifecycle.md`** — Detailed event lifecycle documentation. Structure:

1. **Overview**: The bundle's lifecycle has 5 stages: Request -> Resolve -> Boot -> Run -> Clear. Three Symfony events mark the transitions.

2. **Stage 1: Request Interception**
   - `TenantContextOrchestrator` is a `kernel.request` listener at priority 20
   - Priority 20 places it after the Router (32) and before Security (8)
   - Why priority 20: Controllers must receive tenanted services. Security runs at 8 — if tenancy ran after security, the security component would see un-tenanted state.
   - Show the `onKernelRequest` method signature and what it does: calls `ResolverChain::resolve($request)`.

3. **Stage 2: Resolution**
   - `ResolverChain::resolve()` iterates registered resolvers in priority order (descending).
   - Each resolver receives the `Request` and returns `?TenantInterface`.
   - First non-null result wins. Returns `array{tenant: TenantInterface, resolvedBy: string}`.
   - If no resolver matches, `resolve()` returns null — request proceeds without tenant context.
   - `TenantContext::setTenant()` stores the resolved tenant.
   - **Event: `TenantResolved`** dispatched with the tenant and `resolvedBy` identifier.

4. **Stage 3: Bootstrapping**
   - `BootstrapperChain::boot($tenant)` iterates all registered bootstrappers.
   - Each bootstrapper's `boot(TenantInterface)` method is called in tag priority order.
   - The chain records which bootstrappers ran (as FQCNs).
   - **Event: `TenantBootstrapped`** dispatched with the tenant and the list of bootstrapper FQCNs.
   - After this event, the application runs with full tenant context.

5. **Stage 4: Application Execution**
   - Controllers, services, and Doctrine queries all operate in the tenant context.
   - TenantContext is a read-only value holder at this point.

6. **Stage 5: Teardown**
   - `TenantContextOrchestrator::onKernelTerminate()` triggers cleanup.
   - `BootstrapperChain::clear()` runs all bootstrappers' `clear()` method in **reverse** order of `boot()`. Last booted = first cleared.
   - `TenantContext::clear()` removes the active tenant.
   - **Event: `TenantContextCleared`** dispatched (empty signal event, no payload).

7. **Event Reference Table**:

| Event | Payload | When | Use Case |
|-------|---------|------|----------|
| `TenantResolved` | tenant, resolvedBy | After resolver chain succeeds | Logging, audit trail, metrics |
| `TenantBootstrapped` | tenant, bootstrapperFqcns | After all bootstrappers boot | Post-boot hooks, lazy initialization |
| `TenantContextCleared` | (none) | After all bootstrappers clear | Cleanup, connection pooling |

8. **Console Commands**: Note that `ConsoleResolver` operates independently — listens on `ConsoleCommandEvent`, not `kernel.request`. Bootstraps the same chain but triggered differently.

**`docs/architecture/di-compilation.md`** — DI compilation pipeline. Structure:

1. **Overview**: The bundle uses 3 compiler passes to wire services at container compile time. All three run during the `BeforeOptimization` phase.

2. **BootstrapperChainPass**:
   - Collects all services tagged `tenancy.bootstrapper`
   - Sorts by tag priority (default 0)
   - Injects sorted service references into `BootstrapperChain` as an iterable argument
   - Auto-tagging: `registerForAutoconfiguration(TenantBootstrapperInterface::class)` adds the tag automatically

3. **ResolverChainPass**:
   - Collects all services tagged `tenancy.resolver`
   - Sorts by tag priority (default 0, higher = runs first in resolver chain)
   - Injects sorted service references into `ResolverChain`
   - Auto-tagging: `registerForAutoconfiguration(TenantResolverInterface::class)`

4. **MessengerMiddlewarePass**:
   - Guarded by `interface_exists(MessageBusInterface::class)` — skipped entirely when Messenger not installed
   - Runs at priority 1 (before Symfony's MessengerPass at priority 0) because MessengerPass consumes the parameter
   - Reads all configured bus IDs from `messenger.buses.*` container parameters
   - Prepends `TenantSendingMiddleware` and `TenantWorkerMiddleware` to every bus's middleware list
   - Uses direct parameter modification (not `prependExtensionConfig`) because `performNoDeepMerging()` on bus config would overwrite prepended middleware

**`docs/architecture/dbal-wrapper.md`** — TenantConnection wrapperClass internals. Structure:

1. **Overview**: DBAL 4 supports `wrapperClass` — a subclass of `Doctrine\DBAL\Connection` that intercepts the connection object. `TenantConnection` extends `Connection` and provides `switchTenant()` and `reset()` to change connection parameters at runtime without rebuilding the DI container.

2. **Why ReflectionProperty**: DBAL 4 made the `$params` property private on `Connection`. There is no public setter. The only way to mutate connection parameters at runtime is via `ReflectionProperty`. This is an intentional design choice by the bundle — DBAL's internal API is stable and the `params` property has existed since DBAL 2.

3. **switchTenant() Flow**:
   - Receives `array $config` (from `Tenant::getConnectionConfig()`)
   - Merges `$config` over `$this->originalParams` using `array_merge()`
   - Writes merged params to `Connection::$params` via ReflectionProperty
   - Calls `$this->close()` to destroy the current PDO connection
   - Next query triggers lazy reconnect with the new params

4. **reset() Flow**:
   - Restores `$this->originalParams` to `Connection::$params` via ReflectionProperty
   - Calls `$this->close()`

5. **originalParams**: Captured in the constructor — the placeholder connection params from Doctrine config. This is the "factory default" that `reset()` restores.

6. **Thread Safety / Shared State**: The connection object is shared within the DI container. `switchTenant()` mutates shared state — this is by design for single-request PHP. Long-running workers (Messenger, Swoole) must clear context between operations (handled by `TenantWorkerMiddleware`).

7. **Why Not create a new Connection**: Creating a new DBAL Connection would require replacing the DI service reference everywhere. The wrapperClass pattern allows in-place mutation — all services that hold a reference to the tenant connection automatically use the new params on next query.

Use `!!! warning` for the ReflectionProperty note — this is a conscious decision, not a hack.

**`docs/architecture/sql-filter.md`** — TenantAwareFilter internals. Structure:

1. **Overview**: `TenantAwareFilter` is a Doctrine SQL Filter. When enabled, it intercepts every DQL/Query and conditionally appends `WHERE tenant_id = :slug` to entities marked with `#[TenantAware]`.

2. **SQLFilter Base Class**: Doctrine's `SQLFilter` has a final constructor taking `EntityManagerInterface` only. Custom filters use setter injection for additional dependencies. `TenantAwareFilter` uses `setTenantContext()` setter (called by `SharedDriver::boot()`).

3. **addFilterConstraint() Logic**:
   - Receives `ClassMetadata $targetEntity` and `string $targetTableAlias`
   - Checks `$targetEntity->getReflectionClass()->getAttributes(TenantAware::class)`
   - If no `#[TenantAware]` attribute found: returns `''` (no filter applied)
   - If attribute found AND tenant context exists: returns `{alias}.tenant_id = '{slug}'`
   - If attribute found AND NO tenant context:
     - Strict mode ON: throws `TenantMissingException`
     - Strict mode OFF: returns `''` (no filter — all rows returned)

4. **Null Guard**: When `tenantContext` is null (not yet injected), the filter returns `''` silently. This prevents crashes in console commands that boot Doctrine before `SharedDriver::boot()` runs.

5. **SharedDriver::boot()**:
   - Enables the `tenancy_aware` filter on the EntityManager
   - Calls `setTenantContext()` with the TenantContext service
   - Sets `strict_mode` flag from config parameter

6. **SharedDriver::clear() is a No-Op**: Documented intentionally. The filter reads `TenantContext` live on every query — when `TenantContext::clear()` runs (in `BootstrapperChain`), the filter automatically stops scoping. No explicit disable needed.

7. **SQL Output Example**: Show what the generated SQL looks like before and after the filter:
   ```sql
   -- Without filter:
   SELECT i.* FROM invoices i

   -- With filter (tenant 'acme' active):
   SELECT i.* FROM invoices i WHERE i.tenant_id = 'acme'
   ```

Use `!!! danger` admonition for the strict mode OFF data leak risk.

**`docs/architecture/messenger-lifecycle.md`** — Messenger stamp lifecycle. Structure:

1. **Overview**: Tenant context must survive process boundaries. When a message is dispatched in one process and consumed in another (Redis, RabbitMQ, Doctrine transport), the consumer has no HTTP request to resolve from. `TenantStamp` carries the tenant identity across the boundary.

2. **Dispatch Phase (TenantSendingMiddleware)**:
   - Middleware wraps `$stack->next()->handle($envelope, $stack)`
   - Before passing to the next middleware, checks `TenantContext::hasTenant()`
   - If tenant active: `$envelope = $envelope->with(new TenantStamp($tenant->getSlug()))`
   - If no tenant: envelope passes through unchanged (no stamp added)
   - Position: prepended to middleware stack via `MessengerMiddlewarePass` — runs before all user middleware

3. **Serialization Phase**:
   - `TenantStamp` implements `StampInterface` — Symfony's serializer handles it automatically
   - The slug (string) is serialized with the envelope metadata
   - No custom serializer needed — stamps are serialized by the transport layer

4. **Consume Phase (TenantWorkerMiddleware)**:
   - Reads `TenantStamp` from the envelope: `$stamp = $envelope->last(TenantStamp::class)`
   - If no stamp: passes through to next middleware (no tenant context needed)
   - If stamp found:
     - Looks up tenant via `TenantProviderInterface::findBySlug($stamp->getTenantSlug())`
     - Sets tenant in `TenantContext`
     - Calls `BootstrapperChain::boot($tenant)` — full context (DB, cache, etc.) is restored
     - Calls `$stack->next()->handle($envelope, $stack)` in a `try` block
     - `finally` block calls `BootstrapperChain::clear()` then `TenantContext::clear()`

5. **Key Design Decision: No TenantResolved Event**:
   - The worker middleware does NOT dispatch `TenantResolved`
   - Reason: `TenantResolved` is for HTTP resolution — listeners might assume a `Request` object exists
   - The tenant is being *restored* from a stamp, not *resolved* from a request

6. **Sequential Message Isolation**:
   - Worker processes messages sequentially (one at a time)
   - The `try/finally` guarantees cleanup between messages
   - Message A (tenant acme) → boot(acme) → handle → clear → Message B (tenant demo) → boot(demo) → handle → clear
   - No state leaks between messages

7. **Bus Enrollment (MessengerMiddlewarePass)**:
   - Runs at priority 1 (before MessengerPass at 0)
   - Reads all configured bus IDs
   - Prepends both middleware to every bus
   - Uses direct parameter modification (not prependExtensionConfig) because performNoDeepMerging() would overwrite

**`docs/architecture/design-decisions.md`** — Rationale behind key architectural choices. Structure as a list of decisions, each with: What, Why, Alternatives Considered, Trade-offs.

1. **strict_mode defaults to ON**
   - What: Querying `#[TenantAware]` entity without tenant context throws `TenantMissingException`
   - Why: "A data leak across tenants is a security incident, not a config mistake." Fail-closed is safer than fail-open.
   - Alternative: Default OFF (more convenient). Rejected — convenience does not justify the data leak risk.
   - Trade-off: Console commands and admin tooling must explicitly boot tenant context or disable strict mode.

2. **TenantContext is zero-dependency**
   - What: No constructor parameters, no injected services, no circular deps.
   - Why: TenantContext is injected into nearly every bundle service. Any dependency would risk circular references during container compilation. A zero-dep leaf node is always safe.
   - Alternative: Inject EventDispatcher into TenantContext to auto-dispatch on setTenant(). Rejected — creates coupling and circular dep risk.

3. **Bootstrapper clear() runs in reverse order**
   - What: If A, B, C boot in that order, clear runs C, B, A.
   - Why: Later bootstrappers may depend on state set up by earlier ones. Clearing in reverse unwinds the stack safely. Same pattern as middleware stacks and database transactions.
   - Alternative: Clear in original order. Rejected — would require each bootstrapper to be fully independent, which is too restrictive.

4. **kernel.request priority 20**
   - What: TenantContextOrchestrator fires at priority 20.
   - Why: Must be after Router (32) so the route is available, before Security (8) so controllers receive tenanted services. Priority 20 is the sweet spot.
   - Alternative: Priority 0 (after security). Rejected — controller constructors would receive un-tenanted services.

5. **DBAL ReflectionProperty for connection switching**
   - What: `TenantConnection::switchTenant()` uses ReflectionProperty to mutate DBAL 4's private `$params`.
   - Why: DBAL 4 made `$params` private with no public setter. The only alternatives are creating a new Connection object (breaks DI references) or using an event-based approach (DBAL has no suitable event).
   - Trade-off: Couples to DBAL internal layout. Mitigated by the `$params` property being stable since DBAL 2.

6. **Optional Doctrine via class_exists guards**
   - What: All Doctrine imports are guarded by `class_exists()` or `interface_exists()`.
   - Why: The bundle should be installable without Doctrine for non-DB use cases (cache isolation, Messenger context only). Hard imports would cause class-not-found fatal errors.
   - Alternative: Make Doctrine a hard dependency. Rejected — reduces bundle applicability.

7. **interface_exists for Messenger (not class_exists)**
   - What: `interface_exists(MessageBusInterface::class)` guards Messenger wiring.
   - Why: `MessageBusInterface` is an interface — `class_exists()` returns false for interfaces in PHP. This was a bug caught during development (caused silent skip of all Messenger wiring).
   - Lesson: Always use `interface_exists()` for interface checks, `class_exists()` for class checks.

8. **ConsoleResolver operates independently**
   - What: ConsoleResolver listens on `ConsoleCommandEvent`, not through ResolverChain.
   - Why: Console commands have no HTTP Request. The ResolverChain requires a Request object. ConsoleResolver adds `--tenant` to every command and bootstraps context directly.
   - Alternative: Modify ResolverChain to accept nullable Request. Rejected — makes the HTTP path more complex for a single special case.

9. **Messenger middleware: TenantResolved NOT dispatched in worker**
   - What: Worker-side middleware does not fire `TenantResolved` event.
   - Why: The tenant is being *restored*, not *resolved*. HTTP listeners attached to `TenantResolved` may assume a Request object. Firing the event in worker context could cause null reference errors.

10. **EntityManagerResetListener uses resetManager(null)**
    - What: `resetManager()` called with no argument (null), not `'tenant'`.
    - Why: Works in both driver modes. In database-per-tenant mode, the tenant EM is named 'tenant'. In shared-DB mode, there is only one default EM. Passing null resets the default EM, which is correct in both cases.

Use `!!! info` admonitions for the "Lesson" items. Use `!!! warning` for trade-offs.
