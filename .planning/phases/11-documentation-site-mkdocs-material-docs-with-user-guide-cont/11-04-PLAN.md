**`docs/contributor-guide/architecture.md`** — Architecture overview for contributors. Structure:

1. **Event-Driven Bootstrapper Model**: The bundle's architecture in one sentence — "When a tenant is resolved, a chain of bootstrappers reconfigure Symfony services for that tenant." Show the full request lifecycle as a flow diagram (text-based):

   ```
   Request arrives
       |
   Router (priority 32)
       |
   TenantContextOrchestrator (priority 20)
       |
   ResolverChain.resolve(request)
       |-- HostResolver (30)
       |-- HeaderResolver (20)
       |-- QueryParamResolver (10)
       |
   TenantContext.setTenant()
       |
   dispatch(TenantResolved)
       |
   BootstrapperChain.boot(tenant)
       |-- DatabaseSwitchBootstrapper
       |-- DoctrineBootstrapper
       |-- CacheBootstrapper
       |-- SharedDriver (if shared_db)
       |
   dispatch(TenantBootstrapped)
       |
   Controller runs in tenant context
       |
   kernel.terminate
       |
   BootstrapperChain.clear()  [REVERSE order]
       |
   TenantContext.clear()
       |
   dispatch(TenantContextCleared)
   ```

2. **Key Design Principles**:
   - TenantContext is a zero-dependency value holder — no constructor params, no circular deps
   - Bootstrappers are registered via DI tags, not manual wiring
   - `clear()` runs in reverse order of `boot()` — last booted, first cleared
   - Doctrine dependencies are optional — `class_exists`/`interface_exists` guards everywhere

3. **Namespace Map**: List the 18 namespaces with what they contain:
   - `Attribute` — TenantAware
   - `Bootstrapper` — BootstrapperChain, DatabaseSwitchBootstrapper, DoctrineBootstrapper, TenantBootstrapperInterface
   - `Cache` — TenantAwareCacheAdapter
   - `Command` — TenantMigrateCommand, TenantRunCommand
   - `Context` — TenantContext
   - `DBAL` — TenantConnection, TenantConnectionInterface
   - `DependencyInjection/Compiler` — BootstrapperChainPass, ResolverChainPass, MessengerMiddlewarePass
   - `Driver` — SharedDriver, TenantDriverInterface
   - `Entity` — Tenant
   - `Event` — TenantResolved, TenantBootstrapped, TenantContextCleared
   - `EventListener` — TenantContextOrchestrator, EntityManagerResetListener
   - `Exception` — TenantNotFoundException, TenantInactiveException, TenantMissingException
   - `Filter` — TenantAwareFilter
   - `Messenger` — TenantStamp, TenantSendingMiddleware, TenantWorkerMiddleware
   - `Provider` — DoctrineTenantProvider, TenantProviderInterface
   - `Resolver` — HostResolver, HeaderResolver, QueryParamResolver, ConsoleResolver, ResolverChain, TenantResolverInterface
   - `Testing` — InteractsWithTenancy

4. **Compiler Passes**: 3 passes — BootstrapperChainPass (collects tenancy.bootstrapper tags), ResolverChainPass (collects tenancy.resolver tags, sorts by priority), MessengerMiddlewarePass (prepends tenant middleware to all buses, guarded by interface_exists).

5. Link to [Architecture Reference](../architecture/index.md) for deep-dive on each subsystem.

**`docs/contributor-guide/custom-resolver.md`** — How to implement a custom resolver. Structure:

1. **Overview**: The bundle accepts any class implementing `TenantResolverInterface`. It is auto-tagged via `registerForAutoconfiguration`. Priority determines order in the chain (highest first).

2. **Interface**: Show the full interface:
   ```php
   namespace Tenancy\Bundle\Resolver;

   use Symfony\Component\HttpFoundation\Request;
   use Tenancy\Bundle\TenantInterface;

   interface TenantResolverInterface
   {
       public function resolve(Request $request): ?TenantInterface;
   }
   ```

3. **Implementation Example** — PathResolver (resolves from URL path `/t/{slug}/...`):
   Show a complete class with:
   - `declare(strict_types=1)` and namespace
   - Constructor injection of `TenantProviderInterface`
   - `resolve()` method that extracts slug from path, queries provider, returns tenant or null
   - Exception handling: catch `TenantNotFoundException` (return null), let `TenantInactiveException` bubble

4. **Registration**: The class is auto-discovered if in `src/` with autoconfigure. Otherwise, tag manually:
   ```yaml
   services:
       App\Resolver\PathResolver:
           tags:
               - { name: tenancy.resolver, priority: 15 }
   ```

5. **Priority**: Explain the priority system — higher number = runs first. Default resolvers: Host=30, Header=20, QueryParam=10. Choose a priority that makes sense for your resolver. Show that you can also use the `#[AutoconfigureTag]` attribute:
   ```php
   use Symfony\Component\DependencyInjection\Attribute\AutoconfigureTag;

   #[AutoconfigureTag('tenancy.resolver', ['priority' => 15])]
   class PathResolver implements TenantResolverInterface
   ```

6. **Testing**: Show a unit test for the custom resolver using a mock Request and a stub TenantProvider.

**`docs/contributor-guide/custom-bootstrapper.md`** — How to implement a custom bootstrapper. Structure:

1. **Overview**: Bootstrappers reconfigure services when a tenant is resolved. Any class implementing `TenantBootstrapperInterface` is auto-tagged and added to the chain.

2. **Interface**: Show the full interface:
   ```php
   namespace Tenancy\Bundle\Bootstrapper;

   use Tenancy\Bundle\TenantInterface;

   interface TenantBootstrapperInterface
   {
       public function boot(TenantInterface $tenant): void;
       public function clear(): void;
   }
   ```

3. **Implementation Example** — MailerBootstrapper (switches SMTP transport per tenant):
   Show a complete class with:
   - `declare(strict_types=1)` and namespace
   - Constructor injection of the mailer transport (or config service)
   - `boot()` method that reads tenant config and switches transport
   - `clear()` method that restores the default transport
   - `!!! warning` admonition: `clear()` MUST undo everything `boot()` did. The chain runs `clear()` in reverse order.

4. **Registration**: Auto-discovered with autoconfigure. Manual tag:
   ```yaml
   services:
       App\Bootstrapper\MailerBootstrapper:
           tags:
               - { name: tenancy.bootstrapper }
   ```

5. **TenantDriverInterface**: Explain the marker interface — if your bootstrapper is a primary isolation driver (like DatabaseSwitchBootstrapper or SharedDriver), implement `TenantDriverInterface` in addition to `TenantBootstrapperInterface`. This is for semantic identification only (no additional methods).

6. **Lifecycle Guarantees**:
   - `boot()` is called for every tenant switch
   - `clear()` is called on `kernel.terminate` and in test `tearDown()`
   - `clear()` runs in reverse order of `boot()`
   - If `boot()` throws, `clear()` is still called for previously booted bootstrappers

7. **Testing**: Show a unit test for the custom bootstrapper.
