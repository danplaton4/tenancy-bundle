**`docs/user-guide/strict-mode.md`** — Strict mode explanation. Must cover:

1. **What strict mode does**: When `strict_mode: true` (default), querying a `#[TenantAware]` entity without an active tenant context throws `TenantMissingException` instead of silently returning all rows from all tenants.
2. **Why it defaults to ON**: "A data leak across tenants is a security incident, not a config mistake." Use a `!!! danger` admonition.
3. **When to disable**: Internal admin tooling, data export scripts, migration runners that intentionally need cross-tenant access. Use a `!!! warning` admonition explaining the risk.
4. **How to disable**: Show `tenancy.strict_mode: false` in YAML/PHP tabs.
5. **How it works technically**: The `SharedDriver::boot()` passes the strict_mode parameter to `TenantAwareFilter`. When strict mode is on and no tenant context exists, the filter throws `TenantMissingException`. When strict mode is off and no tenant context exists, the filter returns an empty string (no WHERE clause appended), allowing all-tenant queries.
6. **Console commands**: Note that `TenantAwareFilter` returns empty string (no scoping) when `TenantContext` has no tenant and strict mode is off — this is by design for console commands that run before any tenant is resolved.

**`docs/user-guide/resolvers.md`** — Complete resolver documentation. Structure:

1. **Overview**: The resolver chain runs at `kernel.request` priority 20. Each resolver implements `TenantResolverInterface` and is tried in priority order (highest first). The first resolver to return a non-null `TenantInterface` wins. If no resolver matches, the request proceeds without a tenant context.

2. **Resolver Priority Table**:

| Resolver | Priority | Trigger | Config Key |
|----------|----------|---------|------------|
| HostResolver | 30 | Subdomain: `acme.example.com` | `tenancy.host.app_domain` |
| HeaderResolver | 20 | Header: `X-Tenant-ID: acme` | (none) |
| QueryParamResolver | 10 | Query: `?_tenant=acme` | (none) |
| ConsoleResolver | N/A | CLI: `--tenant=acme` | (none) |

3. **HostResolver** section:
   - Extracts the last subdomain segment before the `app_domain` suffix. Example: `api.acme.example.com` with `app_domain: example.com` resolves to `acme`.
   - Configuration: `tenancy.host.app_domain: example.com`
   - Works with both subdomains (`acme.example.com`) and custom domains (when app_domain is null, the full host is used as lookup).
   - Catches `TenantNotFoundException` (returns null, lets next resolver try). Lets `TenantInactiveException` bubble up as a 403/500.

4. **HeaderResolver** section:
   - Reads the `X-Tenant-ID` request header value as the tenant slug.
   - Ideal for API-first apps (mobile, SPA) where subdomain routing is not available.
   - Same exception behavior: catches `TenantNotFoundException`, bubbles `TenantInactiveException`.

5. **QueryParamResolver** section:
   - Reads `_tenant` query parameter. Lowest HTTP priority (10).
   - Use case: debugging, admin preview of tenant data.
   - Use `!!! warning` admonition: "Query param resolving exposes tenant slugs in URLs. Use only for internal/debug tooling, not production API endpoints."

6. **ConsoleResolver** section:
   - Operates independently from the HTTP resolver chain. Listens on `ConsoleCommandEvent`, NOT `kernel.request`.
   - Adds `--tenant=<slug>` option to every console command automatically.
   - Bootstraps full tenant context (database, cache) when the option is provided.
   - NOT part of the `tenancy.resolvers` config array — it is always registered.

7. **Enabling/Disabling Resolvers** section:
   - Show the `tenancy.resolvers` config key (default: `['host', 'header', 'query_param', 'console']`).
   - Show example of disabling query param: `resolvers: ['host', 'header']`

8. **Custom Resolver** section:
   - Show a complete example of implementing `TenantResolverInterface`.
   - Show the interface signature: `public function resolve(Request $request): ?TenantInterface`
   - Explain DI tag priority: `#[AutoconfigureTag('tenancy.resolver', ['priority' => 25])]` or YAML service tag.
   - Show that any class implementing `TenantResolverInterface` is auto-tagged by the bundle.
   - Example: `PathResolver` that reads `/tenant/{slug}/...` from the URL path.

All PHP examples should show full class declarations with `declare(strict_types=1)`, namespace, and use statements.
