**`docs/user-guide/database-per-tenant.md`** — Complete database-per-tenant driver documentation. Structure:

1. **Overview**: Each tenant gets its own database. The bundle switches the DBAL connection at runtime using DBAL 4's `wrapperClass` pattern. Two entity managers are configured: `landlord` (central tenant registry) and `tenant` (runtime-switched). This provides maximum isolation — one tenant's data is physically separate from another's.

2. **Configuration**: Show the full Doctrine + tenancy config required:

   Tenancy config (YAML/PHP tabs):
   ```yaml
   tenancy:
       driver: database_per_tenant
       database:
           enabled: true
   ```

   Doctrine config (YAML/PHP tabs) — show the two entity managers:
   ```yaml
   doctrine:
       dbal:
           default_connection: landlord
           connections:
               landlord:
                   url: '%env(DATABASE_URL)%'
               tenant:
                   url: 'sqlite:///:memory:'  # placeholder — switched at runtime
                   wrapper_class: Tenancy\Bundle\DBAL\TenantConnection
       orm:
           default_entity_manager: landlord
           entity_managers:
               landlord:
                   connection: landlord
                   mappings:
                       App:
                           type: attribute
                           dir: '%kernel.project_dir%/src/Entity/Landlord'
                           prefix: App\Entity\Landlord
               tenant:
                   connection: tenant
                   mappings:
                       AppTenant:
                           type: attribute
                           dir: '%kernel.project_dir%/src/Entity/Tenant'
                           prefix: App\Entity\Tenant
   ```

3. **Tenant Entity**: Show how the `connection_config` JSON field stores per-tenant database credentials. Example Tenant record:
   ```php
   $tenant = new Tenant('acme', 'acme');
   $tenant->setDomain('acme.example.com');
   $tenant->setConnectionConfig([
       'driver' => 'pdo_mysql',
       'host' => '127.0.0.1',
       'dbname' => 'tenant_acme',
       'user' => 'acme_user',
       'password' => 'secret',
   ]);
   ```

4. **How It Works** (technical): When a tenant is resolved, `DatabaseSwitchBootstrapper::boot()` calls `TenantConnection::switchTenant()` which merges the tenant's connection config over the original placeholder params using `ReflectionProperty` on DBAL 4's private `$params` field, then calls `close()` to force lazy reconnect on the next query. On clear, `TenantConnection::reset()` restores original params.

5. **Entity Manager Isolation**: The landlord EM never changes. The tenant EM is reset on every tenant switch via `EntityManagerResetListener` to prevent identity map pollution.

6. **Migrations**: Link to [CLI Commands](cli-commands.md) for `tenancy:migrate`.

Use `!!! tip` for the explanation that the tenant connection URL in Doctrine config is a placeholder.
Use `!!! warning` for noting that `resetManager()` is called on every switch — cached EM references become stale.

**`docs/user-guide/shared-db.md`** — Complete shared-DB driver documentation. Structure:

1. **Overview**: All tenants share one database. A Doctrine SQL filter (`TenantAwareFilter`) automatically appends `WHERE tenant_id = :id` to every query for entities marked with `#[TenantAware]`. Simpler to operate (one DB, one set of migrations) but less isolation.

2. **Configuration** (YAML/PHP tabs):
   ```yaml
   tenancy:
       driver: shared_db
       strict_mode: true  # default
   ```
   Note: Do NOT set `database.enabled: true` — the bundle rejects this combination at compile time.

3. **Marking Entities**: Show the `#[TenantAware]` attribute on a Doctrine entity. The entity MUST have a `tenant_id` column:
   ```php
   use Doctrine\ORM\Mapping as ORM;
   use Tenancy\Bundle\Attribute\TenantAware;

   #[ORM\Entity]
   #[TenantAware]
   class Invoice
   {
       #[ORM\Id]
       #[ORM\GeneratedValue]
       #[ORM\Column]
       private ?int $id = null;

       #[ORM\Column(length: 255)]
       private string $tenantId;

       // ... other fields
   }
   ```

4. **How the SQL Filter Works**: `SharedDriver::boot()` enables the `tenancy_aware` filter on the default entity manager and sets the `tenant_id` parameter to the active tenant's slug. The filter's `addFilterConstraint()` checks if the entity class has the `#[TenantAware]` attribute; if yes, it appends `WHERE {alias}.tenant_id = '{slug}'`. Entities without the attribute are unaffected.

5. **Strict Mode**: When `strict_mode: true` (default), querying a `#[TenantAware]` entity without an active tenant throws `TenantMissingException`. Link to [Strict Mode](strict-mode.md) for details.

6. **Mixed Entities**: Entities without `#[TenantAware]` return full result sets regardless of tenant context (e.g., shared lookup tables, user profiles).

Use `!!! danger` for the data leak warning about strict mode being off.

**`docs/user-guide/cache-isolation.md`** — Cache namespace isolation. Structure:

1. **Overview**: The bundle decorates the `cache.app` pool with `TenantAwareCacheAdapter`, which prefixes cache operations with the active tenant's slug. Cache keys written under Tenant A are invisible to Tenant B. This happens at the namespace level (not key prefix) via Symfony's `withSubNamespace()`.

2. **How It Works**: `TenantAwareCacheAdapter` reads `TenantContext` live on every cache operation — it never caches the `withSubNamespace()` result to prevent stale tenant context. When a tenant is active, all `get()`, `set()`, `delete()` operations go through a sub-namespaced pool. When no tenant is active, operations go through the original pool.

3. **Configuration**: Cache isolation is automatic — no config needed. The `CacheBootstrapper` is registered as a `tenancy.bootstrapper` service.

4. **Cache Clearing**: Clearing Tenant A's cache does not affect Tenant B. Each tenant's cache lives in a separate namespace.

5. **Custom Cache Pools**: If you use custom pools beyond `cache.app`, you need to decorate them manually with `TenantAwareCacheAdapter`.

Use `!!! info` admonition about the automatic registration.

**`docs/user-guide/messenger.md`** — Messenger integration. Structure:

1. **Overview**: When a message is dispatched while a tenant context is active, a `TenantStamp` carrying the tenant slug is automatically attached. When a worker processes the message, the stamp restores full tenant context before the handler runs.

2. **Requirements**: `symfony/messenger` must be installed. The bundle detects it via `interface_exists(MessageBusInterface::class)`. If not installed, all Messenger wiring is silently skipped. Add `symfony/messenger` to your project: `composer require symfony/messenger`.

3. **How It Works** — Three stages:
   - **Dispatch**: `TenantSendingMiddleware` checks if `TenantContext` has a tenant. If yes, it attaches a `TenantStamp` with the tenant slug to the envelope.
   - **Serialize/Transport**: The stamp is serialized with the envelope and travels through the transport (Redis, RabbitMQ, Doctrine, etc.).
   - **Consume**: `TenantWorkerMiddleware` reads the `TenantStamp`, looks up the tenant via `TenantProviderInterface`, boots the full tenant context via `BootstrapperChain::boot()`, runs the handler, then clears context in a `try/finally` block.

4. **Important**: The worker middleware does NOT dispatch `TenantResolved` — the tenant is restored, not resolved. This prevents HTTP-specific listeners from firing in worker context.

5. **Teardown Guarantee**: The `try/finally` in `TenantWorkerMiddleware` ensures cleanup even when the handler throws. Two messages with different tenant stamps processed sequentially get the correct isolated context.

6. **Bus Enrollment**: The `MessengerMiddlewarePass` compiler pass automatically prepends both middleware to every configured message bus.

Use `!!! warning` admonition about the `interface_exists` guard — if you see "stamps not being attached", verify `symfony/messenger` is installed.

**`docs/user-guide/cli-commands.md`** — CLI commands. Structure:

1. **tenancy:migrate**: Runs Doctrine migrations for all tenants (or a single tenant with `--tenant=<slug>`).
   - Requires: `database.enabled: true` AND `doctrine/migrations` installed.
   - Iterates all tenants from the provider, boots tenant context, runs migrations, reports per-tenant results.
   - Does not stop on first error — reports all failures.
   - Usage: `bin/console tenancy:migrate` (all), `bin/console tenancy:migrate --tenant=acme` (one).

2. **tenancy:run**: Wraps any Symfony console command with tenant context.
   - Usage: `bin/console tenancy:run acme "cache:clear"` — runs `bin/console cache:clear --tenant=acme` as a subprocess.
   - The command creates a subprocess (via `symfony/process`) with the inner command, adding the `--tenant` flag. The ConsoleResolver in the child process picks up the tenant.
   - Requires `symfony/process` (production dependency).

Show `!!! tip` about `tenancy:migrate` requiring both `database.enabled: true` and `doctrine/migrations`.

**`docs/user-guide/testing.md`** — Testing with InteractsWithTenancy. Structure:

1. **Overview**: The `InteractsWithTenancy` trait provides `initializeTenant()`, `clearTenant()`, assertion helpers, and automatic `tearDown()` cleanup for PHPUnit `KernelTestCase` tests.

2. **Setup**: Show a complete test class using the trait:
   ```php
   use Symfony\Bundle\FrameworkBundle\Test\KernelTestCase;
   use Tenancy\Bundle\Testing\InteractsWithTenancy;

   class InvoiceRepositoryTest extends KernelTestCase
   {
       use InteractsWithTenancy;

       public function testFindsOnlyTenantInvoices(): void
       {
           $this->initializeTenant('acme');
           $this->assertTenantActive('acme');

           // Create invoices...
           // Query invoices — only 'acme' invoices returned
       }
   }
   ```

3. **How initializeTenant() Works**: Step by step — clears prior context, creates a synthetic Tenant with :memory: SQLite, activates in TenantContext, runs BootstrapperChain::boot(), resets tenant EM, creates schema. Schema MUST be created AFTER boot() because DatabaseSwitchBootstrapper calls close() which destroys :memory: databases.

4. **Available Methods**:
   - `initializeTenant(string $slug)` — set up a clean tenant context with :memory: SQLite
   - `clearTenant()` — clear context and run bootstrapper teardown
   - `assertTenantActive(string $slug)` — assert a specific tenant is active
   - `assertNoTenant()` — assert no tenant is active
   - `getTenantService(string $class)` — retrieve a service from the test container

5. **Automatic Teardown**: `tearDown()` calls `clearTenant()` automatically. Even if setUp or the test throws, PHPUnit runs tearDown.

6. **Two-Tenant Isolation Test**: Show an example testing that two tenants have separate data.

Use `!!! info` about the :memory: SQLite strategy for fast test execution.
